# ct-improve-plan

사용자 입력: $ARGUMENTS

## 역할

- 이 커맨드는 구현 문서가 아니라 판단 문서를 만든다.
- 출력은 반드시 `{project}_improve_plan.md` 한 개만 만든다.
- 구현 상세, 클래스 설계, API 요청/응답 예시까지 한 문서에 다 쓰지 말고 범위와 우선순위 판단에 집중한다.
- 구현 가이드가 따로 필요한 경우 사용자가 요청한 별도 문서로 분리한다.

## 실행 원칙

- 증거 없이 문제를 단정하지 말고 코드, 런타임 설정, 테스트, 빌드·배포 설정을 근거로 기록하라.
- 프로젝트 문서는 기본 근거로 사용하지 마라. 사용자가 명시적으로 포함한 경우에만 참고하고, 코드 근거와 분리해 취급하라.
- 빠른 결론보다 정확한 범위 판단을 우선하라.
- 이 커맨드는 문서 작성 전용이다. 코드를 직접 수정하지 말고 `mvn`, 빌드, 테스트, 애플리케이션 실행 검증도 수행하지 마라.
- 독립 조사 축은 가능하면 병렬로 진행하되, 제약이 있으면 단일 조사로 대체하고 한계를 남겨라.
- 외부 연동, 정산, 락, 배치, 후처리, 보상 처리처럼 부작용이 큰 항목은 기능 설명보다 경계와 실패 영향 위주로 정리하라.
- 문서는 짧고 판단 가능하게 유지하라. 기본 목표는 120줄 안팎이며, 특별한 경우에도 200줄을 넘기지 마라.
- IMP 항목 수는 보통 3~6개 안에서 정리하라.

## 입력 확정

- 프로젝트 식별자 `{project}`를 확정하라.
- 사용자가 별도 경로를 지정하지 않으면 저장 위치를 현재 작업 루트로 사용하라.
- 출력 파일명은 반드시 `{project}_improve_plan.md` 형식을 사용하라.
- 우선순위는 내부 판단으로 정하되 점수는 문서에 노출하지 마라.

## 작업 순서

1. 코드, 런타임 설정, 테스트, 빌드·배포 설정을 조사하고 가능하면 병렬로 근거를 수집하라.
2. 핵심 코드 경로를 정독하고 입력, 분기, 외부 부작용 지점을 확인하라.
3. 영향 범위를 따라가며 이번 라운드 범위와 후속 범위를 분리하라.
4. 개선 후보를 정리하고 `IMP-xxx` 형식의 ID를 부여하라.
5. 우선순위를 내부적으로 확정하고 문서에는 순번과 이모지로 표현하라.
6. `{project}_improve_plan.md`를 작성하라.

## 문서 작성 규칙

- `improve_plan`은 범위와 우선순위를 정하는 판단 문서다.
- 구현 상세, 클래스 설계, API 요청/응답 예시는 본문에 길게 적지 마라.
- 문서 구조는 아래만 유지하라.
  - `🧭 문서 개요`
  - `📌 개선 항목 요약`
  - `🛠 개선 항목 상세`
  - `📎 이번 라운드 정리`
  - `## 🗂 이력관리`
- `문서 개요`에는 아래만 포함하라.
  - 목적
  - 이번 라운드 범위
  - 후속 범위
- `개선 항목 요약`은 표로 쓰지 말고 리스트로 작성하라.
- 요약 항목은 아래 형식을 사용하라.
  - `(1) 🔴 IMP-01 항목명`
  - `착수 조건`
  - `구현 위임 가능 여부`
- `한 줄 결론`은 필요한 경우에만 짧게 쓰고, 기본적으로는 생략하라.
- 우선순위는 제목의 순번과 색상 이모지로 표현하라.
  - `🔴` 이번 라운드 핵심
  - `🟠` 이번 라운드 포함이지만 선행조건 필요
  - `🟡` 후속 또는 중간 우선순위
- 각 IMP 상세 제목도 요약과 같은 형식을 사용하라.
  - `## (1) 🔴 IMP-01 항목명`
- 각 IMP는 아래만 기본 포함하라.
  - `문제 정의`
  - `권장 접근`
  - `범위`
  - `선행 조건`
  - `구현 위임 가능 여부`
  - `근거`
- `영향`, `비범위`, `검증 기준`, `목표 상태`는 필요한 경우에만 선택적으로 추가하라.
- `구현 위임 가능 여부`는 아래 형식 중 하나로 적어라.
  - `✅ 가능`
  - `🟡 추가 확인 필요`
  - `⏸️ 보류`
- 본문에서도 이해에 필요한 파일명, 클래스명, 메서드명은 간단히 적어도 된다.
- 긴 경로, 버전, URL, 명령어는 `근거`에 우선 정리하라.
- `근거`는 표 대신 bullet 리스트를 사용하라.
- `근거`는 아래 형식을 사용하라.
  - `[짧은 파일명:라인](상대경로)`
  - 다음 줄에 확인 내용 한 줄
- `근거`는 너무 길게 늘어놓지 말고 항목별 2~4개 안에서 정리하라.
- `이번 라운드 정리`에는 아래만 포함하라.
  - `🚀 바로 구현`
  - `🕒 후속 범위`
- 문서 맨 끝에는 반드시 `## 🗂 이력관리` 섹션을 두고 `- YYYY-MM-DD: 변경 내용 요약` 형식으로 기록하라.

## 출력 체크리스트

- `{project}_improve_plan.md`가 생성되었는가.
- `🧭 문서 개요`, `📌 개선 항목 요약`, `🛠 개선 항목 상세`, `📎 이번 라운드 정리`, `## 🗂 이력관리` 구조를 지켰는가.
- 모든 항목이 동일한 `IMP-xxx` 체계를 사용하는가.
- 각 IMP 제목에 `(순번) + 색상 이모지`가 들어갔는가.
- `구현 위임 가능 여부`가 `✅`, `🟡`, `⏸️` 형식으로 표시되었는가.
- `근거`가 표가 아니라 bullet 리스트 형식인가.
- 구현 상세가 과도하게 섞여 들어가지 않았는가.
- 문서가 장황하지 않고 이번 라운드 판단에 집중하는가.
- `## 🗂 이력관리` 섹션이 문서 맨 끝에 있는가.

## Completion Gate

- 아래 조건이 모두 충족되기 전에는 작업을 종료하지 마라.
  - `improve_plan` 작성 완료
  - 각 IMP의 `근거` 작성 완료
  - 각 IMP의 `선행 조건`, `구현 위임 가능 여부` 판단 완료
  - 이번 라운드와 후속 범위 구분 완료
  - 완료 체크리스트 재검토 완료

## Final Output

- 최종 보고에는 아래만 포함하라.
  - 생성 파일명
  - 이번 라운드 핵심 항목 수
  - 후속 범위 유무
