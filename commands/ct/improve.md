# Project Improvement Planner

사용자 입력: $ARGUMENTS

## 역할

- 이 커맨드는 조사와 판단에 집중한다.
- 출력은 반드시 `{project}_improve_plan.md` 한 개만 만든다.
- 구현 diff, 파일별 패치, 실행 절차까지 한 번에 쓰지 말고 판단 문서만 완성한다.
- 이후 구현은 사용자가 특정 `IMP-xxx` 항목을 지정해 `/ct:implement`에 맡길 수 있게 문서 구조를 정리한다.

## 실행 원칙

- 증거 없이 문제를 단정하지 말고 코드, 런타임 설정, 테스트, 빌드·배포 설정을 근거로 기록하라.
- 프로젝트 문서는 기본 근거로 사용하지 마라. 사용자가 문서를 명시적으로 검토 대상으로 포함한 경우에만 참고하고, 그때도 코드베이스 근거와 분리해 취급하라.
- 빠른 결론보다 정확한 구조 파악과 근거 검증을 우선하라.
- 지레짐작으로 결론내리지 말고, 추정은 가설로 표시한 뒤 검증 가능한 근거를 확보할 때만 확정하라.
- 독립 조사 축은 가능하면 병렬로 진행하되, 도구나 세션 제약이 있으면 단일 조사로 대체하고 한계를 결과에 남겨라.
- 요청 범위를 넘는 기능 확장을 제안하지 말고 개선 대상과 우선순위를 먼저 고정하라.
- 이 커맨드는 문서 작성 전용이다. 코드를 직접 수정하지 말고 `mvn`, 빌드, 테스트, 애플리케이션 실행 검증도 수행하지 마라.
- 미확정 항목은 삭제하지 말고 `구현 위임 가능 여부`를 `추가 확인 필요` 또는 `보류`로 남겨라.
- 외부 PG, 메시지, 정산, 락, 배치, 후처리처럼 부작용이 있는 흐름은 기능 분리보다 `트랜잭션 경계`, `응답 계약`, `보상 처리`, `멱등성`을 먼저 확정하라.

## 입력 확정

- 프로젝트 식별자 `{project}`를 확정하라.
- 사용자가 별도 경로를 지정하지 않으면 저장 위치를 현재 작업 루트로 사용하라.
- 출력 파일명은 반드시 `{project}_improve_plan.md` 형식을 사용하라.
- 우선순위 산정 기준을 아래 기본값으로 시작하되, 사용자가 기준을 주면 덮어써라.
  - 영향도(1-5), 결함 위험도(1-5), 전파 범위(1-5), 구현 노력(1-5)
  - 기본 점수: `(영향도*2 + 결함위험도*2 + 전파범위) - 구현노력`
  - 기본 등급: P0(>=12), P1(9-11), P2(6-8), P3(<=5)
- 산정 결과는 내부 우선순위 결정에만 사용하고 문서 본문이나 표에는 노출하지 않는다.

## 작업 순서

1. 독립 조사 축(코드, 런타임 설정, 테스트, 빌드·배포 설정)을 분리하고, 가능하면 병렬로 근거를 수집하라.
2. 핵심 코드 경로를 정독하고 입력부터 반환까지 주요 조건, 분기, 외부 부작용 지점을 확정하라.
3. 호출과 전파 경로를 따라가며 영향 범위를 끝까지 확인하라.
4. 설정, 테스트, 빌드·배포 설정을 코드와 대조해 누락, 불일치, 검증 공백을 확인하라.
5. 실제 처리 결과와 응답 코드가 어긋나는 분기, cleanup 실패가 본 처리 결과를 덮어쓰는 분기, 중복 실행 위험이 있는 분기를 별도로 표시하라.
6. 개선 후보를 수집하고 각 항목의 `문제 정의`, `영향`, `목표 상태`, `권장 접근`을 정리하라.
7. 중복과 종속 항목을 정리하고 개선 항목 ID(`IMP-001` 형식)를 부여하라.
8. 우선순위를 계산해 내부 순서를 확정하라.
9. 각 항목마다 `착수 조건`, `선행 조건`, `구현 위임 가능 여부`를 판단하라.
10. `{project}_improve_plan.md`를 작성하라.
11. 완료 체크리스트로 누락을 다시 점검하고 비어 있는 항목을 모두 보완하라.

## 문서 작성 규칙

- `improve_plan`은 판단 문서다. 무엇을 왜 바꿔야 하는지, 무엇을 먼저 바꿔야 하는지를 전략, 리스크, 영향 관점에서 정리하라.
- 문서 전체 구조는 아래를 유지하라.
  - `문서 개요`
  - `개선 항목 요약`
  - `개선 항목 상세`
  - `이번 라운드 정리`
  - `## 이력관리`
- `문서 개요`에는 `목적`, `이번 라운드 범위`, `제외 범위`, `우선순위 기준`, `전제 제약`만 포함하라.
- `개선 항목 요약` 표는 `ID`, `개선 영역`, `한 줄 결론`, `우선순위`, `착수 조건`, `구현 위임 가능 여부` 컬럼만 사용하라.
- `개선 항목 상세`는 항목별로 `문제 정의`, `영향`, `목표 상태`, `권장 접근`, `범위`, `비범위`, `선행 조건`, `검증 기준`, `구현 위임 가능 여부`, `근거 목록`만 포함하라.
- `목표 상태`는 변경 완료 후 시스템이 어떤 상태여야 하는지 적는다. 완료 기준처럼 써라.
- `권장 접근`은 목표 상태에 도달하기 위한 구현 전략을 적는다. 방법과 방향만 쓰고 diff나 세부 패치는 적지 마라.
- `범위`와 `비범위`는 가능하면 파일, 모듈, 기능 경계 단위로 써라.
- `착수 조건`은 다른 IMP 항목 완료, 외부 합의, 환경 준비 등 이 항목에 착수하기 위한 전제를 적는다. 없으면 `즉시 가능`으로 써라.
- `선행 조건`은 구현 전에 먼저 확인하거나 정리해야 하는 기술적·업무적 전제를 적는다.
- `구현 위임 가능 여부`는 `가능`, `추가 확인 필요`, `보류` 중 하나로 기록하라.
- 요약 표의 `구현 위임 가능 여부`와 상세 항목의 값은 반드시 같아야 한다.
- 본문과 표에는 우선순위 점수, 파일명, 코드 식별자, 라이브러리명이나 버전, URL, 테스트나 빌드 명령, diff를 적지 마라.
- 파일 경로, 코드 식별자, 라이브러리와 버전, 명령어 등 추적용 정보는 `근거 목록`에만 정리하라.
- `근거 목록`은 기본적으로 코드, 설정, 테스트, 빌드·배포 파일만 사용하라. 프로젝트 문서는 사용자가 명시적으로 요구한 경우가 아니면 포함하지 마라.
- `근거 목록`은 bullet 나열 대신 Markdown 표로 작성하라. 최소 컬럼은 `구분`, `위치/대상`, `확인 내용`을 사용하라.
- `근거 목록:` 라인 다음에는 반드시 한 줄을 비운 뒤 표를 시작하라.
- `근거 목록`의 `위치/대상`에서 긴 파일 경로는 전체 경로를 그대로 쓰지 말고 파일 기준 상위 2개 디렉터리까지만 남겨라. 그보다 앞 경로는 `../`로 축약하라.
- 라인 번호나 범위가 필요하면 축약 경로 뒤에 공백으로 덧붙여 표기하라.
- 보안과 의존성 항목은 서비스 영향과 심각도를 중심으로 쓰고, 긴급 패치와 중장기 정비를 분리해 판단하라.
- 외부 연동, 데이터 정합성, 정산, 비동기 후처리, cleanup, 대량 배치처럼 실패 영향이 큰 항목은 `권장 접근` 또는 `검증 기준`에 실패 처리 방향을 반드시 남겨라.
- 실패 처리 방향에는 필요에 따라 `실패 시 기본 동작`, `재시도 여부`, `보상 또는 수동 정리 필요 여부`를 포함하라.
- `이번 라운드 정리`에는 `즉시 착수`와 `추가 확인 필요`만 포함하라. `보류`는 요약 표에서만 관리하라.
- 문서 맨 끝에는 반드시 `## 이력관리` 섹션을 두고 `- YYYY-MM-DD: 변경 내용 요약` 형식으로 기록하라.

## 출력 체크리스트

- `{project}_improve_plan.md`가 생성되었는가.
- `문서 개요`, `개선 항목 요약`, `개선 항목 상세`, `이번 라운드 정리`, `## 이력관리` 구조를 지켰는가.
- 모든 항목이 동일한 `IMP-xxx` 체계를 사용하는가.
- 각 항목에 `문제 정의`, `영향`, `목표 상태`, `권장 접근`, `범위`, `비범위`, `선행 조건`, `검증 기준`, `구현 위임 가능 여부`, `근거 목록`이 포함되었는가.
- `착수 조건`이 요약 표에서 짧고 판단 가능한 값으로 작성되었는가.
- `근거 목록`이 Markdown 표 형식으로 정리되었는가.
- `근거 목록`의 `위치/대상`이 `../` 축약 규칙을 지켰는가.
- 요약 표와 상세 항목의 `구현 위임 가능 여부`가 같은가.
- 구현 diff나 파일별 패치가 본문에 섞여 들어가지 않았는가.
- 고위험 항목에 실패 처리 방향이 남아 있는가.
- `## 이력관리` 섹션이 문서 맨 끝에 있는가.

## Completion Gate

- 아래 조건이 모두 충족되기 전에는 작업을 종료하지 마라.
  - `improve_plan` 작성 완료
  - 각 항목의 `근거 목록` 작성 완료
  - 각 항목의 `착수 조건`, `선행 조건`, `구현 위임 가능 여부` 판단 완료
  - 미확정 항목의 `추가 확인 필요` 또는 `보류` 사유 정리 완료
  - 완료 체크리스트 재검토 완료

## Final Output

- 최종 보고에는 아래를 포함하라.
  - 생성 파일 목록
  - 확정 항목 수
  - `추가 확인 필요` 또는 `보류` 항목 수
  - 즉시 착수 가능한 항목 수
  - 남은 누락 없음 여부
