# Project Improvement Planner

사용자 입력: $ARGUMENTS

## 실행 원칙

- 증거 없이 문제를 단정하지 말고 운영 지표, 장애 이력, 보안 공지, 규제/감사 요구를 근거로 기록하라.
- 성능(속도) 최적화보다 충분한 확인 시간을 확보한 정밀 점검을 우선하라.
- 지레짐작으로 결론내리지 말고, 추정은 가설로 표시한 뒤 검증 가능한 근거를 확보할 때만 확정하라.
- 근거 수집/검증 단계에서는 병렬 처리(독립 조사 트랙 동시 수행)를 기본 전략으로 사용하라.
- 요청 범위를 넘는 기능 확장을 제안하지 말고 개선 대상의 우선순위를 먼저 고정하라.
- `overview`는 개발 구현 관점이 아니라 전략/리스크/영향 관점으로 작성하라.
- `overview` 표는 `문제/근거`를 하나의 통합 컬럼으로 작성하라.
- `overview` 문서에는 우선순위(P0~P3/점수)를 표기하지 마라. 우선순위 산정은 내부 작업에만 사용하라.
- `overview`의 본문/표에는 구현 단서(파일 경로, 클래스/메서드명, 라인번호, 명령어, 의존성/라이브러리명, 버전, URL)를 기록하지 마라.
- 참조 가능한 식별 정보(URL, 파일, 코드 식별자, 버전, 명령어)는 문서 하단 `참고/레퍼런스`에만 기록하라.
- `detail`은 개발자가 문서만 보고 바로 적용 가능한 "패치 지시서"로 작성하라.
- `detail`은 `변경 범위`와 `구현 패치(복붙용 diff 블록)`만 담고, 문제 배경/우선순위/사유/검증 설명은 반복하지 마라.
- `detail`의 구현 안내는 문장형 설명을 최소화하고, 복사-붙여넣기 가능한 unified diff 블록을 기본 단위로 제공하라.
- 이 커맨드는 문서 작성 전용이므로 `mvn`/빌드/테스트/애플리케이션 실행 검증을 수행하지 않는다.
- 파일명은 반드시 아래 형식을 그대로 사용하라.
  - `{project}_improvemnt_overview.md`
  - `{project}_improvemnt_detail.md`
- `overview`와 `detail`의 항목 ID를 동일하게 유지하라.

## 입력 확정

- 프로젝트 식별자 `{project}`를 확정하라.
- 사용자가 별도 경로를 지정하지 않으면 저장 위치를 현재 작업 루트로 사용하라.
- 우선순위 산정 기준을 아래 기본값으로 시작하되, 사용자가 기준을 주면 덮어써라.
  - 영향도(1-5), 장애/보안 리스크(1-5), 발생 빈도(1-5), 구현 노력(1-5)
  - 기본 점수: `(영향도*2 + 리스크*2 + 빈도) - 구현노력`
  - 기본 등급: P0(>=12), P1(9-11), P2(6-8), P3(<=5)
- 산정 결과는 내부 우선순위 결정에만 사용하고 문서 본문/표에는 노출하지 않는다.

## 작업 순서

1. 독립 조사 축(코드/설정/테스트/로그/운영 이력/외부 공지)을 분리하고 병렬 처리로 근거를 수집하라.
2. 코드/설정/테스트/로그/운영 이력에서 개선 후보를 수집하라.
3. 각 후보를 전략 관점의 `문제/근거(통합 설명형)`로 요약하라.
4. 중복/종속 항목을 정리하고 개선 항목 ID(`IMP-001` 형식)를 부여하라.
5. 우선순위 점수를 계산하고 내부 우선순위를 확정하라(문서 비표기).
6. `overview` 문서를 작성하라.
7. `detail` 문서를 작성하라.
8. 두 문서 간 ID/근거 설명/참고-레퍼런스 링크 정합성을 검증하라.

## 문서 작성 규칙

- `overview`는 중복 컬럼을 만들지 말고 `문제/근거` 통합 컬럼 중심으로 간결하게 작성하라.
- `overview` 본문/표에는 파일명(예: `pom.xml`), 코드 식별자, 라이브러리명/버전(예: `commons-io 2.5`), URL, 테스트/빌드 명령을 적지 마라.
- URL, 파일 경로, 코드 식별자, 라이브러리/버전, 명령어, 외부 공지 링크 등 참조자료는 문서 하단 `참고/레퍼런스` 섹션에 정리하라.
- 버전/라이브러리 이슈는 `overview`에서 라이브러리명/버전 자체를 노출하지 말고, 서비스 영향과 심각도(Critical/High/Medium/Low)만 기술하라.
- 보안 관련 개선 항목은 문서 하단 `참고/레퍼런스`에 `CVE-ID`와 확인 가능한 `CVE 경로(URL)`를 반드시 함께 명시하라. 둘 중 하나라도 없으면 해당 항목을 `검증 필요`로 표시하라.
- `detail`은 항목별로 아래만 포함하라.
  - 변경 범위(수정 파일/클래스/설정 키/데이터 경계)
  - 구현 패치(복붙용 diff 블록)
- `detail`에는 문장형 일반론(예: "정리", "개선", "보완")을 쓰지 말고, 파일 단위 diff 블록을 제공하라.
- diff 블록은 반드시 fenced code block(` ```diff `)을 사용하라.
- diff 블록은 unified diff 형식을 사용하고, 파일 헤더(`---`, `+++`)와 hunk(`@@`)를 포함하라.
- 각 diff 블록은 가능한 한 그대로 붙여넣어 `git apply` 가능한 형태로 제공하라.
- 다중 파일 변경은 파일별 diff 블록으로 분리하라.
- 근거가 불충분하면 단정하지 말고 `검증 필요`로 표시하고, diff 대신 `검증 필요` 표기만 남겨라.

## 출력 체크리스트

- 두 파일이 모두 생성되었는가.
- `overview`의 각 항목이 `detail`에 1:1 이상으로 매핑되는가.
- `overview` 표가 `문제/근거` 통합 컬럼 중심 최소 컬럼만 사용했는가.
- `overview` 문서에 우선순위(P0~P3/점수)가 표기되지 않았는가.
- `overview` 하단에 `참고/레퍼런스` 섹션이 별도로 존재하는가.
- `overview` 본문/표에 개발 단서가 없고, 참조 정보가 `참고/레퍼런스`로 분리되었는가.
- 보안 관련 항목의 `CVE-ID`와 `CVE 경로(URL)`가 `참고/레퍼런스`에 모두 포함되었는가(누락 시 `검증 필요` 표기).
- `detail` 각 항목이 `변경 범위`와 `구현 패치(복붙용 diff 블록)`만 포함하는가.
- `detail` diff 블록이 ` ```diff ` + `---/+++` + `@@` 형식을 만족하는가.
- `detail`에 `overview` 내용(문제/근거/우선순위/배경) 반복이 없는가.
- 미검증 가정이 남아 있으면 `검증 필요`와 검증 계획이 명시되었는가.

## 내장 템플릿

### Overview 최소 컬럼

- `ID`
- `개선 영역`
- `문제/근거`

### Overview 하단 필수 섹션

- `참고/레퍼런스`
  - 내부 근거(운영 지표/장애 이력/감사 지적)
  - 외부 근거(보안 공지/CVE/장애 보고서/감사 요구)
  - 보안 이슈는 `CVE-ID`, `CVE 경로(URL)`, `서비스 영향/노출 범위`를 함께 기록
  - URL, 파일명, 코드 식별자, 라이브러리명/버전, 명령어를 포함해 추적 가능한 참조 정보를 기록한다

### Detail 최소 컬럼 (개발 가이드형)

- `ID`
- `변경 범위`
- `구현 패치(복붙용 diff 블록)`

## Detail 작성 예시 형식 (항목별)

- `ID`: IMP-00X
- `변경 범위`: 수정 대상 파일/클래스/설정 키를 명시
- `구현 패치(복붙용 diff 블록)`:

```diff
--- a/src/main/java/com/example/service/SampleService.java
+++ b/src/main/java/com/example/service/SampleService.java
@@ -1,7 +1,11 @@
 public class SampleService {
     public String normalize(String input) {
-        return input.trim();
+        if (input == null) {
+            return "";
+        }
+        return input.trim();
     }
 }
```
