# ct-design-review

사용자 입력: $ARGUMENTS

## 실행 원칙

- 사용자 요청을 바로 전문가 의견으로 흩트리지 말고, 먼저 공통 브리프를 짧게 정리하라.
- 기능 요약과 기능 성격 분류를 분리하라. 요약은 사실 정리이고, 성격 분류는 검토 전략 선택 근거다.
- 전문가 후보군은 고정하되 실제 호출 대상은 요청 성격에 따라 동적으로 선정하라.
- 독립 검토는 가능하면 병렬로 진행하되, 세션 정책상 Agent 위임이 허용될 때만 병렬 Agent를 사용하라.
- Agent 위임이 불가능하면 같은 단계 구조를 유지한 채 직접 전문가별 의견을 순차 작성하라.
- 최종 문서는 토론 로그가 아니라 의사결정 결과물로 작성하라.
- 고위험 설계안은 구현 순서만 적지 말고, 왜 이 순서가 필요한지와 각 단계가 다음 단계의 위험을 어떻게 줄이는지까지 설명하라.
- 외부 연동, 데이터 정합성, 정산, 비동기 후처리, cleanup, 대량 배치처럼 실패 영향이 큰 설계안은 운영 모델 선택지와 권장안을 함께 정리하라.
- 고위험 설계안은 전문가 의견을 단순 찬반으로 끝내지 말고, 어떤 산출물이 먼저 확정돼야 하는지까지 적어라.
- 출시를 나눠야 하는 안건은 구현 순서만 적지 말고, 릴리스 단계별 목적과 각 단계의 성공 기준을 함께 정리하라.
- 제품 영향이 큰 안건은 기술 성공 여부만이 아니라 운영 비용, 사고 예방, 대응 시간 단축 같은 제품 관점 성공 기준도 함께 확인하라.
- 문서 본문에는 이전 초안이나 수정 과정을 남기지 말고 최종안만 기록하라.
- 문서 마지막에는 반드시 `## 이력관리` 섹션을 두고 `- YYYY-MM-DD: 변경 내용 요약` 형식으로 마감하라.

## 전문가 후보군

| 전문가 | 기본 역할 |
|--------|-----------|
| product | 사용자 가치, 범위, 우선순위 판단 |
| domain | 업무 규칙, 정책, 현업 맥락 검토 |
| backend | 서버 구조, 데이터/API 흐름, 확장성 검토 |
| frontend-ux | 사용자 흐름, 화면 구조, 입력/오류 경험 검토 |
| qa | 테스트 전략, 회귀 위험, 검증 기준 검토 |
| security | 권한, 민감정보, 오남용 위험 검토 |
| sre | 배포, 장애 영향, 운영 복잡도 검토 |
| tech-lead | 충돌 정리와 최종 결론 도출 |

## 전문가 역할 정의

### product

**역할**: 사용자 가치, 문제 정의, MVP 범위, 우선순위를 판단한다.

**판단 기준**:
- 이 기능이 해결하는 사용자 문제는 명확한가.
- 지금 범위에서 반드시 필요한 것과 후속으로 미뤄도 되는 것이 구분되는가.
- 구현 비용 대비 효과가 충분한가.
- 단계적 출시가 필요한 안건인지, 한 번에 내보내도 되는 안건인지 구분되는가.
- 단계적으로 출시한다면 각 단계의 목표와 성공 기준이 제품 관점에서 설명되는가.
- 이 설계가 운영 비용, 사고 예방, 대응 시간 단축, 사용자 혼선 감소에 어떤 영향을 주는가.

**피해야 할 것**:
- 세부 기술 설계에 과도하게 끌려가 범위 판단을 흐리지 말 것.
- 모든 요구를 한 번에 넣으려 하지 말 것.

**출력 형식**: 판단 / 근거 / 우려사항 / 권장안 / 제품 관점 성공 기준 / 후속 검토 필요 사항

---

### domain

**역할**: 업무 규칙, 정책, 현업 운영 맥락, 예외 처리 기준을 검토한다.

**판단 기준**:
- 도메인 규칙과 충돌하는 동작이 있는가.
- 정책상 예외 케이스가 빠져 있지 않은가.
- 현업 운영 방식과 맞지 않는 흐름이 없는가.
- 성공, 재시도, 수동 정리, 중복 요청 같은 정책 판정이 한 기준으로 일관되게 설명되는가.
- 부분 성공이나 예외 상황이 정책적으로 어떻게 판정되는지 한 번에 정리할 수 있는가.

**피해야 할 것**:
- 일반적인 기술 관점만 반복하지 말 것.
- 명확한 규칙 없이 추정으로 정책을 확정하지 말 것.

**출력 형식**: 판단 / 근거 / 우려사항 / 권장안 / 필요한 판정표 / 후속 검토 필요 사항

---

### backend

**역할**: 서버 구조, 데이터 모델, API 계약, 상태 전이, 확장성과 유지보수성을 검토한다.

**판단 기준**:
- 구현 구조가 단순하고 일관적인가.
- 데이터 흐름과 상태 전이가 명확한가.
- 기존 시스템 영향과 확장 부담이 관리 가능한가.
- 상태 저장 구조와 상태 전이 규칙이 먼저 확정돼야 하는가.
- 멱등 처리, 재시도, 재처리 방식이 설계 수준에서 구분돼 있는가.
- API 계약과 내부 처리 책임 경계가 섞이지 않는가.

**피해야 할 것**:
- 과도한 미래 대비 설계로 MVP를 무겁게 만들지 말 것.
- 화면 요구를 그대로 서버 책임으로 흡수하지 말 것.

**출력 형식**: 판단 / 근거 / 우려사항 / 권장안 / 대안과 제외 이유 / 필요한 선행 산출물 / 후속 검토 필요 사항

---

### frontend-ux

**역할**: 사용자 흐름, 화면 구성, 입력 경험, 오류 메시지, 학습 비용을 검토한다.

**판단 기준**:
- 사용자가 한 번에 이해할 수 있는 흐름인가.
- 입력/오류/빈 상태가 자연스럽게 처리되는가.
- 화면 복잡도에 비해 얻는 가치가 충분한가.

**피해야 할 것**:
- 내부 구조 편의만으로 UI를 복잡하게 만들지 말 것.
- 시각 요소만 보고 실제 사용 흐름을 놓치지 말 것.

**출력 형식**: 판단 / 근거 / 우려사항 / 후속 검토 필요 사항

---

### qa

**역할**: 테스트 전략, 회귀 위험, 경계 조건, 검증 기준을 검토한다.

**판단 기준**:
- 요구사항이 테스트 가능한 수준으로 정의되어 있는가.
- 상태 변화와 예외 흐름이 검증 가능하게 드러나는가.
- 기존 기능 회귀 위험이 높은 구간이 식별되는가.
- 이 설계의 성공 여부를 어떤 기준으로 판단할 수 있는가.
- 회귀가 발생했을 때 가장 먼저 드러나는 시나리오가 무엇인가.
- 최소 검증 게이트 없이 진행하면 어떤 위험이 생기는가.

**피해야 할 것**:
- 화면 유무만으로 QA 필요 여부를 판단하지 말 것.
- 테스트 항목 나열로 끝내지 말고 검증 기준을 남길 것.

**출력 형식**: 판단 / 근거 / 우려사항 / 후속 검토 필요 사항

---

### security

**역할**: 권한, 인증, 민감정보 처리, 입력 검증, 오남용 가능성을 검토한다.

**판단 기준**:
- 권한 경계가 분명한가.
- 민감정보가 불필요하게 노출되거나 저장되지 않는가.
- 악의적 입력이나 우회 경로가 열려 있지 않은가.

**피해야 할 것**:
- 보안 우려를 추상적으로만 쓰지 말 것.
- 사용자 경험과 운영 비용을 무시한 채 전면 차단만 제안하지 말 것.

**출력 형식**: 판단 / 근거 / 우려사항 / 후속 검토 필요 사항

---

### sre

**역할**: 배포 영향, 운영 복잡도, 장애 전파 범위, 관찰 가능성, 롤백 난이도를 검토한다.

**판단 기준**:
- 배포 시 운영 리스크가 관리 가능한가.
- 장애가 발생했을 때 탐지와 복구가 가능한가.
- 외부 연동과 트래픽 변화가 기존 운영 체계에 무리를 주지 않는가.
- 실패 시 기본 동작과 운영 개입 방식이 명확한가.
- 재시도, 재처리, 수동 정리 중 어떤 운영 모델이 적절한가.
- 장애 전파 범위와 관측 포인트가 설계에 반영돼 있는가.

**피해야 할 것**:
- 기능 자체 타당성보다 운영 편의만 우선하지 말 것.
- 모니터링, 로그, 롤백 계획 없이 안전하다고 단정하지 말 것.

**출력 형식**: 판단 / 근거 / 우려사항 / 운영 모델 권장안 / 필요한 관측 포인트 / 후속 검토 필요 사항

---

### tech-lead

**역할**: 전문가 의견을 종합해 충돌을 정리하고 최종 결론과 권장 구현 방향을 확정한다.

**판단 기준**:
- 사용자 가치, 구현 현실성, 품질 리스크, 운영 리스크 사이의 균형이 맞는가.
- 채택안과 제외안의 경계가 명확한가.
- 남은 이슈가 다음 단계와 혼동되지 않는가.
- 채택안의 실행 순서가 현실적이고 각 단계의 이유가 설명되는가.
- 제외안이나 보류안이 왜 탈락했는지 기록으로 남는가.
- 성공 기준과 재검토 기준이 충분히 명확한가.

**피해야 할 것**:
- 전문가 의견을 단순 요약만 하고 선택 근거를 남기지 말 것.
- 합의가 없는 항목을 확정된 결론처럼 쓰지 말 것.

**출력 형식**: 충돌 지점 / 조율 결과 / 채택안과 제외안의 이유 / 최종 결론 / 권장 구현 방향 / 성공 기준 / 재검토 기준 / 남은 이슈

## 전문가 선정 규칙

### 1차 전문가 기본 선택

| 기능 성격 | 기본 1차 전문가 |
|-----------|-----------------|
| 일반 기능 검토 | product, backend |
| 업무 규칙이 중요한 기능 | product, domain, backend |
| 신규 화면/사용자 흐름 포함 | product, frontend-ux, backend |
| API/데이터 구조 중심 | product, backend |
| 보안/인증이 핵심인 기능 | product, backend, security |
| 외부 연동이 핵심인 기능 | product, backend, sre |
| 운영 영향이 큰 기능 | product, backend |

### 후행 검토 트리거

| 조건 | 후행 검토 대상 |
|------|----------------|
| 개인정보, 인증, 권한, 결제, 민감정보 처리 포함 | security |
| 외부 연동, 배치, 큐, 캐시, 트래픽 변화, 장애 전파 우려 | sre |
| 신규 사용자 흐름, 상태 전이 변경, 회귀 위험 큰 수정, 비동기 처리 추가 | qa |
| 업무 정책 해석 충돌, 규정/정책 의존 로직 | domain |

### 선정 절차

1. 기능 요약에서 입력, 출력, 주요 동작 범위를 정리하라.
2. 기능 성격 분류에서 아래 태그 중 필요한 것을 고르라: `crud`, `workflow`, `ui`, `api`, `policy`, `security-sensitive`, `ops-sensitive`, `external-integration`
3. 성격 태그를 근거로 1차 전문가를 선정하라.
4. 후행 검토 트리거가 하나라도 맞으면 해당 전문가를 추가하라. 단, 이미 1차에 포함된 전문가는 후행에서 제외하라.
5. 항상 `tech-lead`를 마지막 조정자로 포함하라.

## 작업 순서

### 1. 프리패스

- 사용자 요청을 바탕으로 아래 항목을 먼저 확정하라.
  - 검토 대상
  - 요청 배경과 목표
  - 기능 요약
  - 기능 성격 분류
- 기능 성격 분류 결과를 근거로 1차 전문가와 후행 전문가 후보를 선정하라.
- 초기 핵심 쟁점을 3~7개로 정리하라.
- 사용자가 별도 쟁점을 준 경우에는 그대로 반영하고, 누락된 쟁점만 보완하라.

### 2. 1차 전문가 검토

- 기획, 도메인, 백엔드, 프론트엔드/UX처럼 독립 검토가 가능한 전문가를 먼저 투입하라.
- 1차 검토는 서로의 답변을 보지 않은 상태에서 작성하는 것을 우선하라.
- 각 전문가 의견은 아래 4개 항목으로 고정하라.
  - 판단
  - 근거
  - 우려사항
  - 후속 검토 필요 사항
- 고위험 설계안에서는 아래 항목을 추가하라.
  - 권장안
  - 대안과 제외 이유
  - 필요한 선행 산출물

### 3. 2차 후행 검토

- QA, 보안, SRE처럼 선행 의견이나 설계 초안이 있어야 검토 품질이 올라가는 역할은 후행 검토로 분리하라.
- 후행 검토 여부는 감으로 정하지 말고 위의 후행 검토 트리거 규칙을 따르라.
- 후행 전문가는 1차 의견과 핵심 쟁점을 검토한 뒤 추가 리스크와 검증 포인트를 보강하라.

### 4. 조정 및 결론

- 기술 리드 관점으로 의견 충돌 지점을 먼저 정리하라.
- 단순 다수결로 결론내리지 말고, 사용자 가치, 구현 현실성, 품질 리스크, 운영 영향을 함께 비교하라.
- 외부 연동, 정합성, 운영 복잡도가 큰 안건은 조율 결과에 운영 모델 권장안과 제외한 대안의 이유를 함께 남겨라.
- 출시를 단계적으로 가져가는 안건은 최종 결론에 단계 구분의 이유와 각 단계의 목표를 함께 남겨라.
- 최종 결론에는 아래를 반드시 포함하라.
  - 채택안
  - 제외안 또는 보류안
  - 선택 이유
  - 남은 이슈
- 최종 결론에는 채택안의 이유뿐 아니라 성공 기준과 재검토 기준도 함께 정리하라.

### 5. 문서 작성

- 최종 결과는 Markdown 문서로 저장하라.
- 사용자가 파일명을 지정하지 않으면 현재 작업 경로에 아래 우선순위로 저장하라.
  1. `{topic}_design_review.md`
  2. 제목을 안정적으로 슬러그화하기 어렵다면 `design_review.md`
- 파일 저장 후 터미널에는 `검토 대상`, `참여 전문가`, `최종 결론` 3개 항목만 요약해 출력하라.
- 섹션 순서는 반드시 아래를 따르라.
  1. 검토 대상
  2. 요청 배경과 목표
  3. 기능 요약
  4. 기능 성격 분류
  5. 참여 전문가
  6. 핵심 쟁점
  7. 전문가별 의견
  8. 충돌 지점과 조율 결과
  9. 최종 결론
  10. 권장 구현 방향
  11. 남은 이슈
  12. 다음 단계
  13. 이력관리
- `권장 구현 방향`에는 구현 순서만 적지 말고, 각 단계의 이유를 함께 적어라.
- 고위험 설계안의 `권장 구현 방향`에는 `운영 모델 권장안`을 포함하라.
- 단계적 출시가 필요한 안건의 `권장 구현 방향`에는 릴리스 단계 구분과 각 단계의 성공 기준을 포함하라.
- 정책 판정이 핵심인 안건은 `남은 이슈`나 `다음 단계`에 흩어 적지 말고, 별도 판정표 산출물 필요 여부를 명시하라.
- `남은 이슈`에는 미결정 정책, 운영 모델, 외부 합의를 적고, `다음 단계`에는 실제 실행 액션만 적어라.

## 참여 전문가 작성 규칙

- `참여 전문가` 섹션에는 표를 사용하라.
- 최소 컬럼은 아래 4개를 유지하라.
  - 전문가
  - 역할 요약
  - 선정 근거
  - 검토 단계
- `검토 단계`는 `1차`, `후행`, `최종 조정` 중 하나로 적어라.

## 핵심 쟁점 작성 규칙

- 프리패스에서 도출한 쟁점과 검토 중 새로 드러난 쟁점을 함께 관리하라.
- 각 쟁점에는 가능하면 `초기` 또는 `추가` 표시를 붙여라.
- 쟁점은 질문형보다 판단이 필요한 문장형으로 정리하라.

## 종료 체크리스트

- 전문가 선정 근거가 기능 성격 분류와 연결되는가.
- 참여 전문가 표에 단계 구분이 보이는가.
- 핵심 쟁점이 초기/추가 구분을 반영하는가.
- 충돌 지점과 조율 결과가 단순 의견 요약이 아니라 실제 선택 근거를 담는가.
- `권장 구현 방향`이 MVP 범위, 제외 범위, 우선순위를 담는가.
- 고위험 설계안에 운영 모델 권장안과 제외한 대안의 이유가 남아 있는가.
- `권장 구현 방향`에 구현 순서뿐 아니라 각 단계의 이유가 포함되어 있는가.
- `최종 결론` 또는 `권장 구현 방향`에 성공 기준과 재검토 기준이 포함되어 있는가.
- 전문가 의견이 단순 의견 요약에 그치지 않고 필요한 선행 산출물을 남기고 있는가.
- 단계적 출시가 필요한 안건에 릴리스 단계와 각 단계의 성공 기준이 포함되어 있는가.
- 정책 판정이 핵심인 안건에 판정표 형태의 후속 산출물이 필요한지 명확히 남겼는가.
- `남은 이슈`와 `다음 단계`가 분리되어 있는가.
- 터미널에 `검토 대상`, `참여 전문가`, `최종 결론` 요약이 출력됐는가.
- 문서 끝에 `## 이력관리`가 있는가.
