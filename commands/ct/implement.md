# CT Implement

사용자 입력: $ARGUMENTS

## Overview

- 먼저 입력 내용을 읽고 실제 개발 요청으로 바로 실행할 수 있는지 판단하라.
- 입력은 간단한 기능 설명, 개발문서, 개선 계획 문서, 설계 검토 문서까지 포함할 수 있다.
- 입력이 구현 명세로 바로 충분하지 않으면, 실행 가능 범위와 미결정 사항을 먼저 정리해 결론을 안내하라.
- 사용자가 진행을 승인하면, 같은 컨텍스트를 유지한 채 다시 처음부터 해석하지 말고 바로 구현 단계로 이어가라.
- 구현만 수행하지 말고, 연관 영향 수정까지 함께 완료하라.
- 테스트/빌드/실행 검증은 기본 범위가 아니지만, 검증 미실행에 따른 위험은 최종 보고에 반드시 남겨라.

## Execution Principles

- 판단은 반드시 이번 실행에서 확인한 근거로만 수행하라.
- 이전 대화 기억이나 추정만으로 범위를 확정하지 말라.
- 입력의 성격을 먼저 구분하라:
  - 기능 설명
  - 개발문서
  - 계획 문서
  - 설계 검토 문서
- 계획 문서와 설계 검토 문서는 그대로 구현 명세로 취급하지 말라.
- 설계 검토 문서의 '최종 결론'과 '권장 구현 방향'은 확정된 방향으로 취급하고, '남은 이슈'는 미결정 항목으로 분류하라.
- 구현 전에 아래 3가지를 먼저 정리하라:
  - 확정된 요구사항
  - 합리적 가정으로 진행 가능한 항목
  - 미결정 정책 또는 추가 확인이 필요한 항목
- 미결정 정책이 동작 계약, 상태 모델, 응답 계약, 보상 처리에 직접 영향을 주면 임의로 구현하지 말라.
- 모든 핵심 판단은 `파일:라인` 또는 실행한 검색 결과로 역추적 가능해야 한다.
- 구현 완료 판단은 "코드 반영 + 전파 확인" 2요소가 모두 충족될 때만 가능하다.

## Mandatory Steps

- 요청을 받으면 아래 흐름을 기본 수행하라.
  1. 입력 내용 해석
  2. 실행 가능 여부 판단 및 결론 안내
  3. 사용자 승인 대기 (반드시 멈추고 기다려라)
  4. 승인 후 구현
  5. 영향범위 동시 수정
  6. 결과 보고
- 결론 안내 후에는 반드시 멈추고 사용자 응답을 기다려라. 사용자가 명시적으로 진행을 승인하기 전까지 구현 단계로 이동하지 말라.
- "개발 진행해", "구현해", "바로 해줘" 같은 즉시 실행 요청이어도, 입력이 계획 문서나 설계 검토 문서인 경우에는 반드시 먼저 결론을 안내하고 승인을 기다려라.
- 사용자가 검토만 요청한 경우에는 구현하지 말고 결론 안내에서 종료하라.
- 사용자가 명시적으로 제외하지 않는 한 일부 단계만 수행하고 종료하지 말라.

## Workflow

0. 입력 해석 및 실행 가능성 판단
- 요청 또는 문서를 읽고 이번 작업의 성격을 분류하라.
- 구현 가능한 범위를 아래 형식으로 정리하라:
  - 이번에 바로 구현할 항목
  - 가정을 두고 구현 가능한 항목
  - 정책 미확정으로 보류할 항목
- 복수 문서 간 결론이 다를 경우, 어느 쪽을 따랐는지 판단 근거를 함께 안내하라.
- 결론은 짧고 분명하게 안내하라.
- 사용자가 진행을 승인하면, 같은 컨텍스트를 유지한 채 다음 단계로 바로 이동하라.

1. 요구사항 확정
- 구현 승인 후에는 앞 단계에서 정리한 확정 요구사항과 허용된 가정을 기준으로 범위를 고정하라.
- 승인 전 검토에서 이미 정리한 내용은 다시 처음부터 해석하지 말라.
- 요청에서 기능, 제약, 예외를 추출하라.
- 모호하면 최소 질문 1개만 하고, 가능하면 가정을 명시한 뒤 진행하라.
- 큰 개선 항목이면 이번 실행에서 실제로 반영할 구현 단위로 범위를 축소하라.
- 요청을 아래 변경 유형으로 분류하라(복수 선택 가능):
  - 정책 추가/변경
  - 검증 강화/완화
  - 데이터 필드 추가/변경
  - 반환/예외/로그 계약 변경

2. 영향 범위 탐색 (병렬)
- 아래 범위를 병렬로 탐색하라:
  - 대상 메서드/클래스
  - 동일 책임의 형제 메서드
  - 호출 체인(상위/하위 호출자, 동기/비동기 실행 경로)
  - 데이터 경계(DTO/VO/Mapper/XML/SQL/응답 객체)
- 탐색 결과를 "영향범위 매트릭스"로 정리하라:
  - 항목
  - 적용 여부(적용/미적용/검토필요)
  - 근거(`파일:라인` 또는 검색 결과)
  - 조치 내용

3. 구현
- 대상 코드에 요구사항을 반영하라.
- 동일 정책이 필요한 연관 경로도 함께 반영하라.
- 필요 시 공통 검증 메서드나 공통 처리 구조로 정리하라.
- 호환성을 우선하라. 공통/검증 헬퍼 메서드는 DTO/VO 객체 전체 전달보다 필요한 필드 인자 전달을 우선하라.
- 기존 반환 계약, 에러 처리, 로그 패턴은 사용자가 변경을 요구하지 않는 한 유지하라.

4. 관련 범위 수정
- 신규 조건/필드로 인한 누락 전파를 수정하라.
- 호출부의 처리 계약(성공/실패/예외) 일관성을 보정하라.
- 보조 실행 경로(배치/스케줄/운영 API 등) 반영 여부를 명확히 하라.
- Mapper alias, VO 필드, Builder 세팅 정합성을 보정하라.
- 조건식이 DTO/VO 필드를 참조하면, 해당 필드의 생성/전달 경로를 모두 점검하고 누락을 보정하라.
- 연관 경로를 미적용으로 판단한 경우, 미적용 근거를 매트릭스에 명시하라.

5. 결과 보고
- 아래 순서로 요약하라:
  - 구현 내용
  - 영향범위 수정 내용
  - 미적용/보류 항목과 근거
  - 검증 미실행 또는 정책 보류에 따른 잔여 위험
- 영향범위 매트릭스를 포함해 적용/미적용과 근거를 함께 보고하라.

6. 차이 보완 후 종료
- 사용자와 합의한 세부 규칙(로그 문구, 순서, 반환 메시지, 메서드 형태)을 마지막에 재검증하라.
- 차이가 남으면 즉시 코드에 반영하고, "남은 차이 없음" 상태를 확인한 뒤 종료하라.

## Completion Gate

- 아래 항목 중 하나라도 누락되면 작업을 종료하지 않는다.
  1. 입력 해석 및 실행 가능 여부 판단 완료
  2. 요구사항 구현 반영
  3. 영향범위 매트릭스 작성 및 근거 첨부
  4. 호출체인/데이터경계 전파 보정 또는 미적용 근거 명시
  5. 검증 미실행 또는 정책 보류 위험 명시
  6. 최종 차이 점검 결과 "남은 차이 없음" 확인

## Review Output Format

- 검토 단계 최종 응답에는 아래를 포함하라.
  - 입력 성격 분류
  - 구현 가능 여부 결론
  - 이번에 바로 구현 가능한 범위
  - 가정이 필요한 범위
  - 정책 미확정으로 보류할 범위
  - 진행 시 우선 수정할 파일/영역

## Final Output Format

- 최종 응답에는 아래를 포함하라.
  - 변경 파일 목록
  - 입력 해석 결과(기능 설명/개발문서/계획 문서/설계 검토 문서)
  - 이번 구현 범위와 제외 범위
  - 변경 유형 분류 결과
  - 핵심 구현 포인트
  - 영향범위 수정 포인트(매트릭스 요약: 적용/미적용/근거)
  - 미적용/보류 항목과 근거
  - 검증 미실행 항목과 잔여 위험
  - 최종 차이 보완 결과(남은 차이 유무)

## Implementation Checklist

- [ ] 입력 해석 및 실행 가능 여부 판단 완료
- [ ] 요구사항 구현 완료
- [ ] 변경 유형 분류 완료
- [ ] 영향범위 매트릭스 작성 완료
- [ ] 연관 범위(호출체인/데이터경계/보조 실행 경로) 수정 완료
- [ ] 전파 누락 점검(필드 생성/전달 경로) 완료
- [ ] 미적용/보류 항목과 위험 명시 완료
- [ ] 합의사항 대비 차이 보완 완료
