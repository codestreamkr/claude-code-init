# CT Implement

사용자 입력: $ARGUMENTS

## Overview

- 목표는 "한 번의 지시로 끝내는 구현"이다.
- 구현만 수행하지 말고, 연관 영향 수정까지 함께 완료한다.
- 테스트/빌드/실행 검증은 이 커맨드의 범위가 아니다.
- 중간에 사용자가 추가 지시하지 않아도 아래 단계를 자동으로 수행한다.

## Execution Principles

- 판단은 반드시 "이번 실행에서 확인한 근거"로만 수행한다.
- 이전 대화 기억/추정으로 범위를 확정하지 않는다.
- 모든 핵심 판단은 `파일:라인` 또는 실행한 검색 결과로 역추적 가능해야 한다.
- 구현 완료 판단은 "코드 반영 + 전파 확인" 2요소가 모두 충족될 때만 가능하다.

## Mandatory Steps

- 구현 요청을 받으면 아래 3단계를 기본 수행하라.
  1. 기능 구현
  2. 영향범위 동시 수정
  3. 결과 보고
- 사용자가 명시적으로 제외하지 않는 한 일부 단계만 수행하고 종료하지 말라.

## Workflow

1. 요구사항 확정
- 요청에서 기능/제약/예외를 추출한다.
- 모호하면 최소 질문 1개만 하고, 가능하면 가정 명시 후 진행한다.
- 요청을 아래 변경 유형으로 분류한다(복수 선택 가능):
  - 정책 추가/변경
  - 검증 강화/완화
  - 데이터 필드 추가/변경
  - 반환/예외/로그 계약 변경

2. 영향 범위 탐색 (병렬)
- 대상 메서드/클래스
- 동일 책임의 형제 메서드
- 호출 체인(상위/하위 호출자, 동기/비동기 실행 경로)
- 데이터 경계(DTO/VO/Mapper/XML/SQL/응답 객체)
- 탐색은 병렬 실행한다.
- 탐색 결과를 "영향범위 매트릭스"로 정리한다:
  - 항목
  - 적용 여부(적용/미적용/검토필요)
  - 근거(`파일:라인` 또는 검색 결과)
  - 조치 내용

3. 구현
- 대상 코드 반영
- 동일 정책이 필요한 연관 경로 동시 반영
- 필요 시 공통 검증 메서드로 리팩토링
- 호환성 우선: 공통/검증 헬퍼 메서드는 DTO/VO 객체 전체 전달보다 필요한 필드 인자 전달을 우선한다.
- 기존 반환 계약/에러 처리/로그 패턴을 유지

4. 관련 범위 수정
- 신규 조건/필드로 인한 누락 전파 수정
- 호출부의 처리 계약(성공/실패/예외) 일관성 보정
- 보조 실행 경로(배치/스케줄/운영 API 등) 반영 여부 명확화
- Mapper alias, VO 필드, Builder 세팅 정합성 보정
- 조건식이 DTO/VO 필드를 참조하면, 해당 필드의 생성/전달 경로를 모두 점검하고 누락을 보정한다.
- 연관 경로를 미적용으로 판단한 경우, 미적용 근거를 매트릭스에 명시한다.

5. 결과를 보고한다.
- 아래 순서로 요약한다.
  - 구현 내용
  - 영향범위 수정 내용
- 영향범위 매트릭스를 포함해 "적용/미적용과 근거"를 함께 보고한다.

6. 차이 보완 후 종료
- 사용자와 합의한 세부 규칙(로그 문구/순서/반환 메시지/메서드 형태)을 마지막에 재검증한다.
- 차이가 남으면 즉시 코드에 반영하고, "남은 차이 없음" 상태를 확인한 뒤 종료한다.

## Completion Gate

- 아래 항목 중 하나라도 누락되면 작업을 종료하지 않는다.
  1. 요구사항 구현 반영
  2. 영향범위 매트릭스 작성 및 근거 첨부
  3. 호출체인/데이터경계 전파 보정 또는 미적용 근거 명시
  4. 최종 차이 점검 결과 "남은 차이 없음" 확인

## Final Output Format

- 최종 응답에는 아래를 포함하라.
  - 변경 파일 목록
  - 변경 유형 분류 결과
  - 핵심 구현 포인트
  - 영향범위 수정 포인트(매트릭스 요약: 적용/미적용/근거)
  - 최종 차이 보완 결과(남은 차이 유무)

## Implementation Checklist

- [ ] 요구사항 구현 완료
- [ ] 변경 유형 분류 완료
- [ ] 영향범위 매트릭스 작성 완료
- [ ] 연관 범위(호출체인/데이터경계/보조 실행 경로) 수정 완료
- [ ] 전파 누락 점검(필드 생성/전달 경로) 완료
- [ ] 합의사항 대비 차이 보완 완료
