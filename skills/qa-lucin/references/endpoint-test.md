# 엔드포인트 테스트

## 언제 먼저 읽는가

- API 테스트, 엔드포인트 테스트, 상태 코드 검증 요청이 들어왔을 때
- 입력 검증, 권한, 응답 구조가 핵심일 때
- 브라우저 없이 빠르게 회귀를 잡아야 할 때

## 먼저 확인할 것

- 서버 프레임워크와 테스트 스택
- 기존 API 테스트 위치와 파일명 규칙
- 테스트 데이터 준비 방식
- 인증 우회 또는 테스트 인증 방식
- 공통 응답 래퍼 구조
- 예외 처리 방식

## 기본 검증 항목

- 정상 상태 코드와 응답 본문
- 입력 오류 시 상태 코드와 메시지
- 인증 실패와 권한 실패
- 필수 필드 누락
- 잘못된 파라미터 형식
- 빈 결과와 예외 흐름

## 놓치기 쉬운 검증 항목

- 멱등성이 필요한 작업에서 같은 요청을 반복해도 결과가 일관되는지
- 중복 요청이나 동시 요청이 들어왔을 때 상태가 꼬이지 않는지
- 페이지네이션과 정렬 파라미터의 경계값이 안전한지
- 빈 결과와 대용량 결과에서도 응답 스키마가 유지되는지
- 기본값이 있는 파라미터를 생략했을 때 계약이 흔들리지 않는지

## 작성 원칙

- 프로젝트에 이미 있는 테스트 스타일을 따른다.
- 상태 코드만 보지 말고 응답 필드와 의미까지 검증한다.
- 권한이 있으면 최소한 성공 케이스와 실패 케이스를 둘 다 둔다.
- 한 테스트에서 너무 많은 정책을 동시에 검증하지 않는다.
- 서비스 내부 구현보다 외부 계약을 우선 검증한다.

## 엔드포인트 테스트 요청에 필요한 정보

- 대상 엔드포인트 경로
- HTTP 메서드
- 인증 필요 여부
- 대표 요청 예시
- 기대 상태 코드
- 기대 응답 구조 또는 필드

## 요청과 질문 기준

- 좋은 요청과 애매한 요청 예시는 `references/request-patterns.md`를 기준으로 본다.
- 정보가 부족하면 먼저 코드에서 경로, 인증 방식, 기존 테스트 구조를 찾고, 그래도 확인되지 않는 항목만 묻는다.
- 직접 실행이 어려울 때의 질문 기준도 `references/request-patterns.md`를 따른다.
