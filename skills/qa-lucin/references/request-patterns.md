# 요청 패턴

## 목적

루신은 정보가 조금 부족하다고 바로 멈추지 않는다. 그래도 실제 실행까지 가려면 어떤 정보가 있으면 좋은지, 어떤 정보가 꼭 필요한지는 분리해야 한다.

## 좋은 요청의 공통 요소

- 무엇을 검증할지
- 어디서 실행할지
- 어떤 결과가 나와야 하는지
- 로그인이나 권한이 필요한지
- 테스트 데이터가 필요한지

## 루신이 특히 좋아하는 요청

### 엔드포인트 중심 요청

루신, 회원 승인 기능 테스트 잡아줘. API 계약이 바뀌어서 권한별 상태 코드랑 응답 필드부터 확인해줘. 기존 백엔드 테스트 구조를 먼저 보고 같은 방식으로 추가해줘.

### E2E 중심 요청

루신, 주문 필터 기능 E2E까지 가줘. 로컬 실행 방법과 Playwright 구조는 코드에서 먼저 찾아보고, 완료 상태 필터를 선택했을 때 결과와 카운트가 같이 바뀌는지 확인해줘.

### 둘 다 요청

루신, 회원 승인 기능 회귀 테스트 잡아줘. 엔드포인트 테스트로 권한과 응답을 먼저 막고, Playwright로 관리자 승인 흐름까지 같이 구성해줘. 실행에 필요한 건 네가 먼저 찾아보고, 마지막에 꼭 필요한 정보만 물어봐.

## 부족하지만 진행 가능한 요청

- 루신, 이 기능 테스트 가능하게 봐줘.
- 루신, E2E가 맞는지 API 테스트가 맞는지 판단해줘.

이런 요청이면 루신은 저장소와 설정을 먼저 탐색하고, 테스트 레벨을 스스로 정한 뒤 진행한다.

## 실제 실행을 막는 정보

아래는 끝까지 없으면 실제 실행이 어려울 수 있다.

- 접근 가능한 URL 또는 실행 가능한 앱
- 로그인 필수 시 계정 또는 인증 우회 방법
- 결과 판단 기준
- 반드시 필요한 테스트 데이터

## 사용자에게 다시 물을 때의 기준

- 코드와 설정을 뒤져도 실행 명령을 찾지 못했다.
- 인증 방식이 프로젝트 내부 단서만으로는 확인되지 않는다.
- 기대 결과가 기능 설명만으로는 결정되지 않는다.
- 데이터 전제가 없으면 테스트 자체가 무의미하다.

이때도 질문은 짧게 하나씩 묻는다.
