# 결함 분석

## 목적

테스트 실패를 봤을 때 무엇이 문제인지 빠르게 분류한다. 루신은 실패를 봤다고 바로 제품 버그로 단정하지 않는다.

## 기본 분류

### 제품 결함

- 기능 동작이 요구사항과 다르다.
- 상태 코드, 응답 구조, 화면 반영이 기대와 다르다.
- 사용자 흐름이 중간에 끊긴다.

### 테스트 코드 결함

- 잘못된 locator, 잘못된 기대값, 잘못된 fixture 사용
- 실제 기능은 정상이지만 테스트가 잘못 짜였다.
- 구현 변경에 맞춰 테스트가 갱신되지 않았다.

### 환경 또는 데이터 불안정

- 테스트 데이터가 준비되지 않았다.
- 외부 의존성이 불안정하다.
- 세션, 네트워크, 시간차 이슈로 결과가 흔들린다.

## 심각도 기준

- 치명: 서비스 핵심 흐름이 막히거나 데이터 손상, 금전 영향, 권한 누락이 있다.
- 높음: 주요 기능 사용이 어렵거나 운영 대응이 바로 필요하다.
- 보통: 우회는 가능하지만 기능 완성도나 신뢰도에 영향을 준다.
- 낮음: 영향 범위가 작고 일정 조건에서만 드러난다.

## 재현 빈도 기준

- 항상: 같은 조건이면 반복 실행해도 계속 재현된다.
- 가끔: 같은 조건처럼 보여도 재현 여부가 흔들린다.
- 한 번: 현재는 한 번만 관찰됐고 재현 조건이 아직 좁혀지지 않았다.

## 우선순위 판단 기준

- 사용자 영향 범위가 얼마나 큰가
- 데이터 손상, 금전, 권한 문제로 이어지는가
- 우회 방법이 있는가
- 재현 빈도가 높은가
- 배포 직전 또는 운영 중 문제인가

## 불안정(flaky) 판단 기준

- 같은 코드와 같은 데이터인데 결과가 흔들린다.
- 재실행하면 통과와 실패가 섞인다.
- 고정 대기나 순서 의존이 들어가 있다.
- 비동기 완료 기준이 명확하지 않다.

불안정(flaky) 이슈는 통과로 보지 않는다. 원인과 재현 조건을 먼저 적는다.

## 결함 정리 방식

- 재현 절차
- 실제 결과
- 기대 결과
- 심각도
- 재현 빈도
- 우선순위
- 영향 범위
- 임시 우회 가능 여부
- 테스트 수준에서 먼저 막을 수 있는지

## 정리 예시

### 제품 결함 예시

- 재현 절차: 관리자 로그인 후 회원 승인 화면에서 승인 버튼 클릭
- 실제 결과: 목록 상태가 그대로 대기 중으로 남음
- 기대 결과: 상태가 승인 완료로 즉시 바뀜
- 영향 범위: 운영자가 승인 결과를 화면에서 바로 확인할 수 없음

### 테스트 결함 예시

- 재현 절차: 주문 필터 E2E 실행
- 실제 결과: 완료 상태 버튼 locator를 찾지 못함
- 원인: 버튼 텍스트가 변경됐는데 테스트가 갱신되지 않음
