# Playwright E2E 테스트

## 언제 먼저 읽는가

- Playwright, E2E, 브라우저 기반 검증 요청이 들어왔을 때
- 실제 사용자 흐름, 화면 상태 변화, 프론트-백 연동을 검증해야 할 때

## 먼저 확인할 것

- `playwright.config.*` 위치와 실행 명령
- 기존 E2E 테스트 디렉터리 구조
- fixture, helper, page object 사용 여부
- 로그인 세션 재사용 방식(`storageState` 포함)
- `data-testid` 사용 여부
- 테스트 데이터 준비 방식
- 브라우저와 baseURL 설정

## 기본 원칙

- role/name 기반 locator를 우선한다.
- `waitForTimeout` 같은 고정 대기는 피한다.
- 한 테스트에는 사용자 의도 하나를 담는다.
- assertion은 사용자가 체감하는 결과를 우선한다.
- 취약한 CSS selector는 마지막 수단으로만 쓴다.
- 기존 프로젝트 구조가 있으면 반드시 따른다.

## 최소 검증 항목

- 대표 정상 흐름
- 주요 상태 변화
- 저장 또는 제출 후 반영 여부
- 오류 메시지 또는 실패 흐름
- 권한 또는 메뉴 가시성 차이

## 요청에 필요한 정보

- 대상 환경 또는 URL
- 로컬 실행 명령 또는 접근 가능한 서버
- 로그인 필요 여부와 계정
- 검증할 화면 흐름
- 기대 결과
- 필요한 테스트 데이터

## 요청과 질문 기준

- 좋은 요청과 애매한 요청 예시는 `references/request-patterns.md`를 기준으로 본다.
- 정보가 부족하면 저장소에서 실행 방법과 화면 흐름을 먼저 찾고, 그래도 막히는 최소 정보만 사용자에게 확인한다.

## 불안정(flaky)을 피하는 기준

- 고정 시간 대기를 넣지 않는다.
- 네트워크 완료보다 화면 상태 완료를 기준으로 대기한다.
- 셀렉터를 DOM 구조에 과하게 의존하지 않는다.
- 재시도보다 원인 분석을 우선한다.

## 실전 보강 항목

- 로그인 비용이 큰 흐름이면 `storageState`로 세션을 재사용할 수 있는지 먼저 본다.
- 응답 완료 자체가 검증 포인트일 때만 `waitForResponse`를 쓰고, 기본은 화면 상태가 바뀌는 시점을 기준으로 대기한다.
- 애니메이션, 실시간 데이터, 브라우저 차이가 큰 화면은 시각 회귀 비교 대상에서 제외한다.
- 병렬 실행을 전제로 테스트 간 데이터와 상태가 서로 간섭하지 않게 만들고, 고정 데이터 ID나 공유 계정 상태에 과하게 의존하지 않는다.
- 가능하면 테스트가 직접 필요한 데이터를 만들거나 충돌 없는 고유 데이터를 사용해 독립성을 유지한다.
