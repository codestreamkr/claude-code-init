# 테스트 전략

## 목적

기능 요청을 받았을 때 어떤 레벨의 테스트를 먼저 둘지 판단한다. 루신은 무조건 E2E부터 가지 않는다. 빠르게 실패를 잡을 수 있는 레벨과 실제 사용자 흐름을 검증해야 하는 레벨을 나눠서 본다.

## 기본 원칙

- 회귀 위험이 큰 지점을 먼저 막는다.
- 같은 위험을 더 싸게 막을 수 있으면 낮은 레벨 테스트를 우선한다.
- 사용자 흐름에서만 드러나는 문제는 E2E로 본다.
- 상태 코드, 권한, 응답 구조는 엔드포인트 테스트로 먼저 고정한다.
- 영향이 큰 기능은 엔드포인트와 E2E를 같이 둔다.
- 둘 다 필요한 경우의 자세한 순서는 아래 섹션을 따른다.

## 우선순위 판단 기준

### 위험이 큰 기능

- 생성, 수정, 삭제, 승인, 결제, 정산
- 권한에 따라 결과가 달라지는 기능
- 화면과 API가 동시에 바뀐 기능
- 장애 재발 이력이 있는 기능

이 경우 둘 다 검토한다.

### 엔드포인트 테스트가 더 효율적인 경우

- 입력 검증이 복잡하다.
- 상태 코드 조합이 많다.
- 권한 조합이 많다.
- 브라우저 없이 빠르게 확인할 수 있다.
- 프론트보다 백엔드 계약 안정화가 먼저다.

### E2E 테스트가 더 필요한 경우

- 로그인부터 완료까지 흐름이 중요하다.
- 필터, 폼, 모달, 테이블, 라우팅, 세션이 같이 움직인다.
- 브라우저 상호작용에서만 문제를 볼 수 있다.
- 프론트와 백엔드가 붙었을 때만 오류가 드러난다.

## 선택 규칙

### 엔드포인트만 두는 경우

- API 계약 검증이 핵심이고, UI 흐름 위험은 낮다.
- 빠른 회귀 보호가 더 중요하다.

### E2E만 두는 경우

- 내부 API가 이미 안정적이고, 실제 사용자 흐름 검증이 핵심이다.
- 화면 상호작용과 상태 변화가 주요 리스크다.

### 둘 다 두는 경우

- 기능 영향이 크다.
- API와 UI가 함께 바뀌었다.
- 같은 장애가 다시 나올 가능성이 높다.
- 낮은 레벨과 높은 레벨에서 모두 의미 있는 실패 지점이 있다.

## 둘 다 필요한 경우의 순서

1. 엔드포인트 테스트로 상태 코드, 권한, 검증 로직, 응답 구조를 먼저 고정한다.
2. 빠르게 실패해야 하는 계약 지점을 먼저 막는다.
3. 이후 E2E 테스트로 실제 화면 흐름, 상태 반영, 프론트-백 연동을 검증한다.

## 최소 세트 권장안

### 엔드포인트 테스트 최소 세트

- 정상 요청 1건
- 대표 입력 검증 실패 1건 이상
- 권한 실패 1건 이상
- 주요 응답 필드 검증

### E2E 테스트 최소 세트

- 대표 정상 흐름 1건
- 대표 실패 흐름 1건 또는 상태 변화 검증 1건
- 저장 후 화면 반영 확인

## 결과 정리 방식

- 왜 이 레벨을 선택했는지 분명히 적는다.
- 자동화에서 제외한 항목은 이유를 남긴다.
- 직접 실행하지 못한 항목은 남은 위험으로 분리한다.
