# 외부 연동 아키텍트 스킬

이 스킬은 **외부 시스템 연동 설계 작업의 공통 뼈대**를 제공한다. 2단계 워크플로우(리포트 → 제안)의 절차·문서 구조·금지 규칙·벤더 레퍼런스 선택 기준을 고정한다. 구현은 이 스킬의 범위가 아니다.

## 이 스킬이 다루는 영역

**다루는 것**
- 외부 연동 현황 분석 절차
- 목표 구조 설계 문서 템플릿
- 전환 순서·롤아웃 체크리스트
- 벤더 레퍼런스 로딩 규칙
- 웹훅·멱등·재처리·재조정 공통 관심사

**다루지 않는 것**
- Spring 구현 문법·저장소 컨벤션·트랜잭션 코드
- SQL 튜닝·인덱스 설계·락·배치 최적화
- 보안(JWT/OAuth2) 구현 세부
- 프론트엔드·UI 영역

**경계 원칙**
- 제안 문서가 완성되면 이 스킬의 역할은 끝난다. 구현은 사용자가 별도 에이전트에게 요청한다
- 데이터 정합성·SQL 성능·락 이슈가 감지되면 해당 영역 전문가 합류를 요청한다
- 설계 문서는 아키텍트가 단독 소유한다. 구현자는 다시 쓰지 않고 코드·PR로 말한다. 변경 요청은 새 제안 문서로 남긴다

---

## 2단계 워크플로우 — 산출물은 분리한다

| 단계 | 목적 | 산출물 | 수정 범위 | 다음 단계 조건 |
|---|---|---|---|---|
| **1. 리포트** | 현재 구조 이해 | 분석 리포트 문서 | **읽기만**. 코드 수정 금지 | 같은 요청에 제안이 포함되어 있으면 바로 제안으로 이어 간다 |
| **2. 제안** | 목표 구조 설계 | 구현 전달용 설계 문서 | **문서만** 작성. 코드 수정 금지 | 제안 완료 후 사용자가 구현 담당 에이전트를 별도 호출 |

**산출물 분리 원칙**: 리포트와 제안은 **문서 섹션이 구분되는 것**이지 **반드시 왕복해야 하는 것**은 아니다. 한 요청에 둘 다 요구되면 한 응답 안에서 리포트 → 제안 순으로 이어 만든다. 단, 리포트 단계에서 리팩터링하거나 제안 단계에서 구현에 착수하는 것처럼 **산출물 성격이 섞이는 것**은 금지다.

---

## 단계별 절차

### Step 1. 리포트

**트리거 예시**
- "기존 KCP 결제 구조 분석해줘"
- "현재 외부 연동 어떻게 되어 있는지 리포트 만들어줘"

**절차**

1. **Phase 0 파악** — 코드·설정·스키마를 읽기만 한다.
   - 빌드 파일(build.gradle / pom.xml), Spring Boot·Java 버전
   - 패키지 구조와 기존 외부 연동 모듈 위치
   - application.yml/properties의 외부 연동 설정
   - 결제 상태 저장 테이블·컬럼
   - 웹훅/콜백 수신 경로
   - 기존 테스트 구조(WireMock·Testcontainers 여부)

2. **호출 흐름 추적** — 진입점부터 벤더 호출·응답 수신·DB 반영까지의 시퀀스를 그린다.

3. **경계 표 작성** — 벤더에 묶인 것과 도메인에 묶인 것을 분리한다.

4. **조인 포인트 식별** — 이관 시 건드려야 할 지점(파일·메서드·설정·테이블).

5. **영향 범위 산정** — 직접 영향·간접 영향·회귀 위험 구역.

6. **리포트 문서 출력** — 아래 템플릿 사용.

**금지 사항**
- 기존 코드 수정, 임의 리팩터링, 공통 추상화 추가
- 목표 구조를 이 단계에서 섞어 제안하기
- 벤더별 구현 디테일에 들어가기(리포트는 **현재** 이해가 목적)

**리포트 문서 템플릿**

```markdown
# [시스템명] 외부 연동 현황 리포트

## 목적
- 이 리포트가 어떤 결정을 지원하는지 한 문단

## 대상 범위 / 비대상
- 대상: 분석한 모듈·흐름
- 비대상: 분석에서 제외한 영역과 이유

## 현재 구조 요약
- 아키텍처 다이어그램/텍스트
- 주요 모듈과 책임

## 호출 흐름
- 요청 진입점 → 벤더 호출 → 응답 수신 → DB 반영 시퀀스
- 콜백/웹훅 경로 별도 기술

## 상태 모델
- 결제/주문 상태 전이도
- 벤더 상태와 도메인 상태의 매핑 현황

## 조인 포인트
| 위치 | 벤더 의존 내용 | 영향 모듈 |
|---|---|---|
| ... | ... | ... |

## 영향 범위
- 직접 영향(파일·메서드·설정·테이블)
- 간접 영향(호출하는 다른 도메인)
- 외부 연동 외 영역(배치·알림·정산 등)

## 리스크
- 기술 부채·안티패턴
- 누락된 멱등성·재처리·관측성

## 회귀 위험
- 이관 시 주의할 지점
- 테스트 커버리지 부족 구역

## 미확인 사항
- 이 리포트에서 확인 못 한 지점과 그 이유
```

### Step 2. 제안

**트리거 예시**
- "리포트 확인하고 토스 이관 제안해줘"
- "목표 구조 설계해줘"

**선행 조건**: 리포트 근거가 확보되어 있어야 한다. 기존 리포트 문서가 있으면 그대로 사용하고, 같은 요청 안에서 방금 만든 리포트가 있으면 그것을 근거로 이어 간다. 둘 다 없으면 짧은 현황 요약으로 근거를 세운 뒤 제안을 잇고, 구조가 복잡해 깊은 분석이 필요할 때만 Step 1을 별도로 요구한다.

**절차**

1. **리포트 재확인** — 리포트의 조인 포인트·영향 범위·리스크를 근거로 삼는다.

2. **벤더 레퍼런스 확인**
   - `references/vendors/{vendor}/*`가 있으면 읽는다(예: `toss/`)
   - 레퍼런스가 없거나 불확실하면 공식 문서나 신뢰 가능한 1차 레퍼런스로 바로 확인한다
   - 핵심 스펙(API·인증·웹훅 서명·에러 맵·멱등 정책)을 정리한다

3. **목표 구조 설계**
   - 포트(도메인 인터페이스) 정의 — 벤더 독립 메서드 시그니처
   - 어댑터(벤더 구현) 분리
   - 상태 전이 재정의 — 도메인 상태와 벤더 상태 분리
   - 오류 분류 체계 — 재시도 가능 / 영구 실패 / 사용자 개입 필요
   - 웹훅·멱등·재처리 구조
   - 관측성(로그 마스킹·추적 ID·메트릭·알림)
   - 시크릿 관리 방식

4. **전환 순서 수립**
   - 스트랭글러 패턴 기본: 기능 단위로 트래픽을 점진 전환
   - 각 단계에 롤백 경로 명시
   - 이관 기간 중 양측 공존 전략(주문별 벤더 식별자 저장 등)

5. **테스트 포인트 정리**
   - 단위·통합·계약·E2E 범위 구분
   - WireMock/Testcontainers 활용 지점

6. **회귀 위험·롤아웃 체크리스트**

7. **구현 전달 문서 출력** — 아래 템플릿 사용. **단일 문서**로 완결한다.

**금지 사항**
- 레거시 구조를 그대로 복제해 목표 구조라 제안
- 공식 문서 미확인 상태에서 필드·상태·오류 코드 단정
- 승인 전 실제 코드 수정
- 이 단계에서 구현 디테일(패키지 경로·트랜잭션 코드)까지 확정하기 — 그건 구현자 몫

**구현 전달 문서 템플릿**

```markdown
# [기능명] 외부 연동 이관 설계 — [KCP → Toss]

## 목적
- 이 이관이 해결하는 문제와 기대 효과

## 변경 대상 / 비대상
- 대상: 변경될 모듈·흐름·테이블
- 비대상: 명시적으로 건드리지 않는 영역

## 현재 구조 요약
- 리포트 참조 + 이 문서에서 필요한 만큼만 요약

## 변경 지점
- 조인 포인트 목록과 변경 방식

## 목표 구조
- 포트·어댑터 구성
- 도메인과 벤더 경계
- 의존 방향 다이어그램

## 클래스 / 인터페이스 책임
| 이름 | 책임 | 의존 |
|---|---|---|
| PaymentGateway (port) | ... | - |
| TossPaymentAdapter | ... | PaymentGateway |
| ... | ... | ... |

## 상태 전이
- 도메인 상태 다이어그램
- 벤더 상태 → 도메인 상태 매핑표

## 오류 처리
- 오류 분류(재시도/영구/사용자개입)
- 벤더 에러 코드 → 도메인 에러 매핑
- 재시도 정책(백오프·최대 횟수·데드레터)

## 웹훅 / 멱등성 / 재처리
- 멱등키 저장소 구조
- 웹훅 서명 검증 방식
- 중복·순서 뒤바뀜·지연 처리
- 재조정 배치 설계

## 구현 순서
1. ...
2. ...
- 각 단계의 롤백 경로

## 테스트 포인트
- 단위 / 통합 / 계약 / E2E 범위
- 핵심 시나리오 목록

## 회귀 위험
- 이관으로 깨질 수 있는 지점
- 완화 전략

## 롤아웃 체크리스트
- [ ] 환경 변수·시크릿 준비
- [ ] 점진 전환 설정
- [ ] 모니터링·알림 연결
- [ ] 롤백 스위치 확인

## 벤더 레퍼런스
- 사용한 레퍼런스 파일 경로
- 참조한 공식 문서 URL
```

제안 문서가 완성되면 아키텍트의 작업은 종료한다. 구현은 사용자가 별도로 구현 담당 에이전트에게 요청한다. 이후 설계 의도에 대한 질문은 답하되 같은 요청 안에서 문서는 다시 쓰지 않는다. 변경 요청이 오면 새 제안 문서를 만든다.

---

## 벤더 레퍼런스 선택 기준

`references/vendors/{vendor}/` 아래 마크다운을 읽는 방식. 아래는 **권장 구성 예시**이며, 실제 파일이 없으면 공식 문서로 바로 간다. 벤더별 레퍼런스를 새로 채울 때 목차로 쓴다.

**Toss 이관 작업**
- `references/vendors/toss/overview.md` — API 전체 지형
- `references/vendors/toss/payment-flows.md` — 일반·간편·빌링·가상계좌·에스크로
- `references/vendors/toss/webhooks.md` — 서명·재전송·이벤트 타입
- `references/vendors/toss/error-codes.md` — 에러 코드 맵
- `references/vendors/toss/migration-from-kcp.md` — KCP ↔ Toss 매핑

**KCP 분석 작업**
- `references/vendors/kcp/legacy-patterns.md` — 기존 연동 해석용

**공통 원칙**
- 레퍼런스는 **요약·길잡이**. 최종 소스는 벤더 공식 문서.
- 구현 단계에서 필드·상태·오류 코드는 공식 문서나 신뢰 가능한 1차 레퍼런스로 한 번 더 확인한다.
- 레퍼런스와 공식 문서가 어긋나면 공식 문서가 우선. 어긋난 지점은 레퍼런스를 갱신한다.

## 외부 연동 공통 관심사 체크리스트

설계 단계에서 이 목록이 전부 다뤄졌는지 확인한다.

- [ ] **포트·어댑터 경계** — 도메인이 벤더를 직접 참조하지 않는가
- [ ] **시크릿 관리** — 환경 변수·Secret Manager·마스킹
- [ ] **타임아웃** — 연결·읽기·전체 타임아웃 각각 설정
- [ ] **재시도** — 멱등한 호출만 재시도, 백오프·최대 횟수
- [ ] **서킷 브레이커** — 장애 전파 차단
- [ ] **멱등성** — 요청 멱등키·수신 측 멱등 검사
- [ ] **웹훅 서명 검증** — 서명 알고리즘·시크릿 로테이션
- [ ] **웹훅 멱등** — 중복 수신·재전송 대비
- [ ] **상태 재조정** — DB·벤더 상태 괴리 감지·보정 배치
- [ ] **오류 분류** — 재시도 / 영구 실패 / 사용자 개입
- [ ] **관측성** — 구조화 로그·추적 ID·메트릭·알림
- [ ] **테스트** — 단위·계약·통합·E2E 역할 분담
- [ ] **롤아웃** — 점진 전환·롤백 스위치
- [ ] **감사** — 이관 전후 비교·정산 일치 검증

## 판단 원리 요약

1. **벤더는 바뀌고 도메인은 남는다** — 경계 고정 우선
2. **큰 바꾸기 = 작은 바꾸기의 합** — 스트랭글러 패턴
3. **외부 호출에 fire-and-forget 없음** — 실패 경로부터 설계
4. **웹훅은 신뢰 불가 채널** — 멱등·재확인 필수
5. **상태는 양쪽에 존재** — 재조정 배치 필수
6. **문서는 하나** — 아키텍트 단독 소유

---

## 금지 사항

1. 단계 혼합(리포트 중 리팩터링, 제안 중 구현 착수)
2. Phase 0 없이 설계 착수
3. 리포트 단계 코드 수정
